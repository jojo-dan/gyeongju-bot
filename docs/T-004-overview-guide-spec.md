## 1. 배경 & 문제 정의

### 현재 상태
- 웹앱(`gyeongju-trip.vercel.app`)은 **일자별 탭 네비게이션**으로 구성
- 하루씩 넘겨봐야 전체 여행 흐름을 파악할 수 있음
- 식당 옵션은 이름 + 메뉴 1줄 + 아빠/히로 뱃지만 표시
- 관광지(불국사 등)는 `desc` 필드 1~2줄이 전부

### 사용자 시나리오
> "내일 부모님한테 링크를 공유하고 설명할 계획. 6일간 갈 곳과 식당을 한눈에 볼 수 있으면 좋겠다."

**핵심 사용자**: 부모님 (50~60대, 모바일 브라우저)
- 링크를 받고 처음 여는 순간, **전체 여행 흐름**을 바로 파악하고 싶음
- 식당을 볼 때: "뭘 파는 곳이고, 맛있는 집인지, 별점은 어떤지" 궁금
- 관광지를 볼 때: "27개월 손녀를 데리고 갈 때 쉴 곳이 있는지, 뭘 꼭 봐야 하는지" 궁금

### 성공 기준
1. 부모님이 링크를 열고 30초 내에 "아, 이런 여행이구나" 파악 가능
2. 식당 이름을 보고 "이 집은 이런 걸 팔고, 별점이 이 정도구나" 파악 가능
3. 불국사 등 주요 장소에 대해 "여기서 뭘 봐야 하고, 아이 데리고 어떻게 다녀야 하는지" 파악 가능

### 3.4 "추천 식당" 결정 로직

식사 시간별 옵션 중 "가장 추천하는 1곳"을 결정하는 기준:

1. `chosen`이 설정된 경우 → 해당 식당이 추천 (확정)
2. `chosen`이 없는 경우 → **옵션 배열의 첫 번째** (봇이 이미 가장 추천순으로 배치)
3. 데이터에 `recommended: true` 필드를 추가하여 명시적으로 지정하는 것도 고려

→ 현재 데이터 구조상 **옵션 배열 순서 = 추천 순서**이므로, 추가 필드 없이 첫 번째 옵션을 추천으로 처리.

### 4.3 데이터 수집 방법

식당 별점/메뉴/가격 정보는 **빌드 타임**에 수집하여 JSON에 포함:

| 데이터 | 소스 | 수집 방법 |
|--------|------|----------|
| 별점 + 리뷰 수 | 네이버지도 | 웹 검색 / 수동 |
| 대표 메뉴 + 가격 | 네이버지도/블로그 | 웹 검색 / 수동 |
| 네이버지도 링크 | 네이버지도 | 식당명으로 검색, URL 복사 |
| 장소 가이드 | 공식사이트 + 블로그 | 웹 리서치 |

> **결정**: 실시간 API 크롤링 대신, **사전 리서치 후 JSON에 하드코딩**한다.
> - 이유: 여행 6일간 식당 목록이 고정적이고, API 인증/비용/속도 이슈 회피
> - 데이터 신선도: 여행 직전 1회 업데이트면 충분

- 기존 index.html(1038줄)에 추가하면 파일이 과도하게 비대해짐
- 별도 URL(`/guide`)로 부모님께 직접 공유 가능
- 같은 `/api/data` 엔드포인트를 공유하므로 데이터 일관성 유지
- Vercel은 정적 파일 라우팅을 자동 처리 (`guide.html` → `/guide`)

**Option B: SPA 내 뷰 전환** (미채택)
- 장점: 데이터 캐시 공유
- 단점: 단일 파일 2000줄 이상, 관심사 분리 어려움, URL 공유 복잡

### 5.3 guide.html 구현 명세

#### 기술 스택
- **프레임워크 없음**: 기존 index.html과 동일하게 바닐라 HTML/CSS/JS
- **디자인 시스템**: 기존 Braun/Dieter Rams 스큐어모피즘 계승
- **폰트**: Playfair Display (serif) + IBM Plex Mono (mono) + System font
- **데이터**: `/api/data` fetch → JSON 파싱 → 렌더링

#### CSS 공유 전략
- 공통 CSS 변수(`:root`)와 기본 컴포넌트 스타일은 guide.html에 복사
- 두 파일 간 CSS 동기화는 수동 (프로젝트 규모상 CSS-in-JS나 빌드 시스템 불필요)
- 향후 필요 시 `styles.css` 공통 파일로 추출 가능

#### 섹션별 렌더링 로직

**① 일정 Overview 섹션**
```javascript
function renderOverview(days) {
  // days 배열을 순회하며 각 day를 카드로 렌더링
  // item.cat에 따라 아이콘 분기: activity→🏛, meal→🍽, cafe→☕
  // chosen이 있으면 "(확정)" 표시
  // optional이면 "선택" 태그 표시
}
```

**② 장소 가이드 섹션**
```javascript
function renderPlaceGuides(days) {
  // cat === 'activity'이고 guide 필드가 있는 아이템만 추출
  // 중복 제거 (동궁과 월지가 Day1, Day2에 모두 있음)
  // guide.mustDo, guide.babyTips, guide.practicalInfo 렌더링
}
```

**③ 식당 가이드 섹션**
```javascript
function renderRestaurantGuide(days) {
  // cat === 'meal' 인 아이템만 추출
  // 일자별·시간별 그룹핑

  // 나머지 options는 접힌(collapsed) 상태로 렌더링
  // accordion 토글: 순수 CSS + <details>/<summary> 또는 JS 토글

```

#### 인터랙션 명세

| 요소 | 동작 | 구현 |
|------|------|------|
| 대안 식당 접기/펼치기 | 탭 시 accordion 토글 | `<details>`/`<summary>` (시맨틱, JS 불필요) |
| 네이버지도 링크 | 탭 시 새 탭에서 네이버지도 열기 | `<a href="..." target="_blank">` |
| Day 카드 → 일정 상세 | 탭 시 기존 앱의 해당 일자로 이동 | `<a href="/?day=N">` + index.html에서 query param 처리 |
| 위로 가기 | 스크롤 시 플로팅 버튼 표시 | `scroll-to-top` 버튼 |

#### 반응형 디자인
- **모바일 우선** (max-width: 420px 기준)
- 부모님이 볼 디바이스: iPhone 또는 Galaxy (모바일 웹브라우저)
- 태블릿/데스크톱에서도 깨지지 않게 max-width 제한

### 5.4 기존 index.html 수정사항

최소한의 변경만 수행:

1. **헤더에 가이드 링크 추가**
```html
<div class="header">
  <h1>경주 가족여행</h1>
  <div class="sub-info">...</div>
  <a href="/guide" class="guide-link">가이드 보기</a>  <!-- 추가 -->
</div>
```

2. **URL query param으로 일자 선택 지원** (선택적)
```javascript
// /guide 에서 "Day 2 상세 보기" 클릭 시 /?day=2로 이동
const params = new URLSearchParams(location.search);
const dayParam = params.get('day');
if (dayParam) activeDay = parseInt(dayParam) - 1;
```

### 5.6 Vercel 라우팅

Vercel은 `webapp/` 디렉토리의 정적 파일을 자동 서빙:
- `webapp/guide.html` → `https://gyeongju-trip.vercel.app/guide`
- 추가 설정 불필요 (vercel.json의 rewrites가 필요할 수 있으나, `.html` 확장자 생략은 Vercel 기본 동작)

### 5.7 성능 고려사항

- **초기 로드**: `/api/data` 1회 fetch (기존과 동일)
- **번들 사이즈**: 단일 HTML 파일, 외부 라이브러리 없음
- **이미지**: 없음 (텍스트 기반 UI)
- **폴링**: 가이드 페이지는 실시간성 불필요 → **폴링 없음** (1회 fetch로 충분)
  - 이유: 가이드는 "참고용 문서"이지 실시간 상태 추적이 아님
